# Часть 1. Vim-основы

## 1. Помощь внутри Neovim

| Хоткей / команда | Действие |
|---|---|
| `:help <тема>` | справка по теме (например `:help split`) |
| `:help` | оглавление справки |
| `:q` | закрыть окно справки |
| `<leader>fh` | fuzzy-поиск по всей справке через Telescope |
| `<leader>fk` | список всех настроенных хоткеев с поиском |
| `<leader>` (подождать) | popup which-key со всеми вариантами после лидера |

**Правило:** забыл хоткей — нажми `<leader>` и подожди секунду. Появится popup со всеми группами. Нажми букву группы (например `f`) — popup перерисуется и покажет варианты `<leader>f...`.

---

## 2. Режимы

Neovim — модальный редактор. В разных режимах одни и те же клавиши делают разное.

| Режим | Назначение | Как войти |
|---|---|---|
| **Normal** | навигация, команды | `<Esc>` (всегда возвращает сюда) |
| **Insert** | обычный набор текста | `i`, `a`, `o` (из normal) |
| **Visual** | выделение текста | `v`, `V`, `<C-v>` (из normal) |
| **Command** | ввод Ex-команды (`:w`, `:Lazy`) | `:` (из normal) |
| **Terminal** | работа во встроенном терминале | автоматически в буфере терминала |

**Главное правило новичка:** если что-то идёт не так — нажми `<Esc>`. Это всегда возвращает в Normal mode.

## 3. Базовая навигация

В Normal mode.

### По символам и словам

| Хоткей | Действие |
|---|---|
| `h` / `j` / `k` / `l` | влево / вниз / вверх / вправо |
| `w` | в начало следующего слова |
| `b` | в начало предыдущего слова |
| `e` | в конец текущего/следующего слова |

### По строке

| Хоткей | Действие |
|---|---|
| `0` | в начало строки (нулевая колонка) |
| `^` | к первому непустому символу строки |
| `$` | в конец строки |
| `f<char>` | прыгнуть к следующему `<char>` в строке |
| `t<char>` | прыгнуть **перед** следующим `<char>` |

### По файлу

| Хоткей | Действие |
|---|---|
| `gg` | в начало файла |
| `G` | в конец файла |
| `<число>G` | на строку с номером (например `42G`) |
| `<C-u>` | вверх на полэкрана (центрировано) |
| `<C-d>` | вниз на полэкрана (центрировано) |
| `<C-b>` / `<C-f>` | на полный экран вверх / вниз |
| `%` | прыгнуть к парной скобке |

### Поиск по файлу

| Хоткей | Действие |
|---|---|
| `/<текст>` | поиск вперёд |
| `?<текст>` | поиск назад |
| `n` / `N` | следующее / предыдущее совпадение |
| `*` / `#` | искать слово под курсором вперёд / назад |
| `<Esc>` (в Normal) | снять подсветку поиска |

| Хоткей | Действие |
|---|---|

| `I` / `A` | вставка в начале / конце строки |

### Удаление и замена

| Хоткей | Действие |
|---|---|

| `cw` | удалить слово и войти в insert |
| `cc` | удалить строку и войти в insert |
| `r<char>` | заменить один символ под курсором |
| `R` | войти в режим замены (overwrite) |

### Копирование и вставка

| Хоткей | Действие |
|---|---|
| `yy` | копировать строку (yank) |
| `yw` | копировать слово |
| `y$` или `Y` | копировать до конца строки |
| `p` / `P` | вставить после / перед курсором |

### Отмена и повтор

| Хоткей | Действие |
|---|---|
| `u` | отменить (undo) |
| `<C-r>` | повторить отменённое (redo) |
| `.` | повторить последнее действие (любое) |

## 6. Окна и сплиты

**Окно (window)** = область экрана с буфером. Экран можно делить на несколько окон.

### Сплиты

| Хоткей / команда | Действие |
|---|---|
| `:split` / `:sp` | горизонтальный сплит |
| `:vsplit` / `:vsp` | вертикальный сплит |
| `<C-w>s` | то же, что `:split` |
| `<C-w>v` | то же, что `:vsplit` |
| `<C-w>q` | закрыть текущее окно |
| `<C-w>o` | оставить только текущее окно |
| `<C-w>=` | выровнять размеры окон |

### Навигация между окнами

| Хоткей | Действие |
|---|---|
| `<C-h>` | в окно слева (например, в neo-tree) |
| `<C-l>` | в окно справа (в код) |
| `<C-j>` / `<C-k>` | вниз / вверх |
| `<C-w>h` / `<C-w>l` | то же, через `<C-w>`-префикс |

### Размеры окон

| Хоткей | Действие |
|---|---|
| `<C-Up>` | высота окна +2 |
| `<C-Down>` | высота окна -2 |
| `<C-Left>` | ширина окна -2 |
| `<C-Right>` | ширина окна +2 |

## 7. Буферы и сохранение

**Буфер** = открытый файл в памяти. Буферов может быть открыто много, окон — обычно меньше.

| Хоткей / команда | Действие |
|---|---|
| `:e <файл>` | открыть файл (или создать новый) |
| `:w` | сохранить |
| `:q` | закрыть текущее окно |
| `:wq` | сохранить и закрыть |
| `:q!` | закрыть без сохранения (потерять изменения) |
| `:wa` | сохранить все открытые файлы |
| `:qa` | закрыть всё |
| `:bd` | закрыть текущий буфер |
| `<S-h>` или `:bp` | предыдущий буфер |
| `<S-l>` или `:bn` | следующий буфер |
| `:ls` / `:buffers` | список всех открытых буферов |

## 10. Workspaces и pinned-проекты

Двухуровневая модель:
- **Workspaces** (БД №1, лимит 20, MRU) — история всех проектов, в которых работал. Каждое открытие `nvim` в новой папке автоматически попадает сюда.
- **Pinned** (БД №2, без лимита) — закреплённые проекты, видны на дашборде.

### Управление

| Хоткей | Действие |
|---|---|
| `<leader>fa` | закрепить текущую папку (pin current cwd) |
| `<leader>fu` | открепить (выбор через `vim.ui.select`) |
| `<leader>fp` | telescope: история workspaces (Enter — открыть; `<C-d>` — удалить из истории) |
| `<leader>fP` | telescope: закреплённые проекты (Enter — открыть; `<C-d>` — открепить) |
| `<leader>fA` | telescope: закрепить проект из истории |

### На дашборде

| Хоткей | Действие |
|---|---|
| `p` | открыть пикер закреплённых проектов (то же, что `<leader>fP`) |
| `1`..`9` | сразу открыть закреплённый проект по номеру |

### Автоматическая реакция на смену каталога

При любом `:cd <путь>` (глобальный — НЕ `:lcd` и `:tcd`):
- закрываются буферы, чьи файлы лежат **вне** нового cwd (буферы `nofile`, дашборда и neo-tree не трогаются);
- если открытых буферов не осталось, создаётся буфер-заглушка `Project: <имя>`;
- cwd добавляется в историю workspaces, MRU обновляется;
- neo-tree автоматически открывается слева с корнем = новый cwd, либо обновляет корень, если уже открыт.

Это значит, что цифры на дашборде, пикеры `<leader>fp`/`<leader>fP` и обычный `:cd` руками — все ведут себя одинаково.

## 11. Neo-tree — файловый эксплорер

Аналог `Ctrl+Shift+E` в VS Code.

### Открытие / закрытие

| Хоткей | Действие |
|---|---|
| `<leader>e` | toggle neo-tree (корень = текущая рабочая директория) |
| `<leader>E` | toggle neo-tree (корень = директория текущего файла) |

### Навигация внутри дерева

| Хоткей | Действие |
|---|---|
| `j` / `k` | вниз / вверх |
| `l` или `<CR>` | открыть файл / развернуть папку |
| `h` | свернуть папку / выйти из неё |
| `<` | подняться на папку выше |
| `.` | сделать текущую папку корнем дерева |
| `H` | показать / скрыть скрытые файлы |
| `R` | обновить дерево |
| `q` / `<Esc>` | закрыть панель |

### Открытие файлов

| Хоткей | Действие |
|---|---|
| `<CR>` или `l` | открыть в текущем окне |
| `v` | открыть в вертикальном сплите справа |
| `s` | открыть в горизонтальном сплите снизу |
| `P` | превью без перехода фокуса |

### Операции с файлами

| Хоткей | Действие |
|---|---|
| `a` | создать файл (введи имя; `/` в конце = папка) |
| `A` | создать папку |
| `r` | переименовать |
| `d` | удалить (с подтверждением) |
| `c` | копировать |
| `m` | переместить |
| `y` / `x` | копировать / вырезать в буфер neo-tree |
| `p` | вставить из буфера neo-tree |
| `Y` | копировать **полный путь** в системный буфер обмена |
| `O` | открыть файл системным приложением (картинки, PDF) |

### Вкладки в шапке neo-tree (клик мышью)

- `File` — обычное дерево файлов.
- `Bufs` — список открытых буферов.
- `Git` — только изменённые файлы.

### Git-статусы у файлов

| Символ | Статус | Описание |
|---|---|---|
| `+` | added | новый файл в git stage (готов к коммиту) |
| `~` | modified | изменён, но не добавлен в stage |
| `✖` | deleted | файл удалён из git |
| `➜` | renamed | переименован / перемещён |
| `★` | untracked | не отслеживается git |
| `◌` | ignored | в `.gitignore` |
| `✗` | unstaged | изменения есть, но не в stage |
| `✓` | staged | в stage, ждёт коммита |
| `` | conflict | конфликт при merge |

Цвета имён файлов также меняются по git-статусу: жёлтый = изменён, зелёный = новый/добавлен, серый = игнорируется.

## 14. Harpoon — закладки на 2-5 файлов

Аналог pinned-табов в VS Code, но для клавиатуры.

| Хоткей | Действие |
|---|---|
| `<leader>ha` | добавить текущий файл в harpoon |
| `<leader>hh` | открыть меню (список + редактирование) |
| `<leader>1`..`<leader>5` | прыгнуть к 1..5-му файлу |
| `<leader>hn` | следующий файл в списке (циклически) |
| `<leader>hp` | предыдущий файл в списке (циклически) |

### В меню harpoon (`<leader>hh`)

- На строке файла → `dd` удаляет файл из списка.
- Порядок строк можно менять обычными vim-командами.
- `<Esc>` или `q` — сохранить и закрыть.

Harpoon хранит список **отдельно для каждого проекта** (определяется по корню `.git`). Список сохраняется между сессиями.

### Поля окна

- `Paths` — где искать (по умолчанию = корень проекта).

## 18. LSP — умный код

> **LSP** — это «умный помощник» для каждого языка, который умеет анализировать код: проверять ошибки в реальном времени, переходить к определению функции, переименовывать всё во всём проекте за одну команду, и т.д.
>
> У нас подключены LSP для языков: **Go** (gopls), **YAML** (yamlls + 700+ схем), **JSON** (jsonls), **TOML** (taplo), **Dockerfile** (dockerls), **docker-compose** (compose-LS), **Lua** (lua_ls), **Bash** (bashls), **Markdown** (marksman).

### Навигация по коду

Все хоткеи работают, когда курсор стоит на каком-то символе (имени функции, переменной, типа).

| Хоткей | Действие |
|---|---|
| `gd` | прыгнуть к месту, где функция/переменная **определена** |
| `gD` | прыгнуть к месту **объявления** (для языков, где это разное; в Go обычно совпадает с `gd`) |
| `gr` | показать **все места**, где используется этот символ во всём проекте |
| `gI` | прыгнуть к **реализации** (для интерфейсов: показать конкретный тип) |
| `gy` | прыгнуть к определению **типа** значения |
| `<C-o>` | вернуться обратно (как «Back» в браузере) |
| `<C-i>` | вперёд по истории прыжков |

### Документация и подсказки

| Хоткей | Где | Действие |
|---|---|---|
| `K` | Normal | На функции — открыть документацию (как Quick Documentation в IDE). На свёрнутом блоке кода — превью содержимого |
| авто, при наборе `(` или `,` | Insert | Рядом с курсором серой подсказкой появляется имя текущего параметра. Например написал `fmt.Fprintf(` — увидишь `▸ w` (имя 1-го параметра). Это `lsp_signature.nvim` |
| `<C-k>` | Insert | Открыть **полное** окно с сигнатурой функции (все параметры, их типы, return value). Рисует blink.cmp в едином стиле с меню автодополнения. Повторный `<C-k>` — закрыть. Само по себе при наборе `(` это окно НЕ всплывает — только по `<C-k>` |

### Изменение кода

| Хоткей | Действие |
|---|---|
| `<leader>la` | **Code action** — меню «что можно сделать с этой строкой»: добавить импорт, переименовать, извлечь функцию, исправить ошибку и т.д. |
| `<leader>lq` | Применить **первый quick-fix** без меню (если есть только один — сразу починит) |
| `<leader>lr` | **Переименовать символ** во всём проекте (rename refactoring) |
| `<leader>lc` | Запустить **code lens** под курсором — это маленькая команда над функцией, например `▶ Run test` над тестом, `▶ Run benchmark` |
| `<leader>li` | Toggle **inlay hints** — серые подсказки рядом с кодом: имена параметров, выведенные типы (для коротких `x :=`), и т.д. По умолчанию выключены |

### Ошибки и предупреждения

| Хоткей | Действие |
|---|---|
| `<leader>ld` | Открыть popup с **полным текстом ошибки** на текущей строке |
| `]d` / `[d` | Прыгнуть к следующей / предыдущей ошибке в файле |
| `<leader>xx`, `<leader>xd` | Открыть **панель со списком всех ошибок** (раздел Trouble, секция 21) |

**Курсор на строке с ошибкой 1-2 секунды** — popup с описанием открывается **автоматически**, без нажатий.

### Внутри popup'ов LSP

В окне hover (`K`), signature help (`<C-k>`), code action (`<leader>la`):

| Хоткей | Действие |
|---|---|
| `q` или `<Esc>` | закрыть popup |
| `j` / `k` | скролл, если содержимое не помещается |
| `<CR>` | в code action — применить выбранное |

## 19. Автодополнение и сниппеты

Движок автодополнения — **blink.cmp**. Аналог IntelliSense в VS Code: при наборе кода всплывает меню вариантов. Сниппеты — **LuaSnip** + библиотека **friendly-snippets** (готовые заготовки кода на десятки языков).

### Меню автодополнения

Меню всплывает **само** при наборе. Источники вариантов: LSP (gopls и др.), пути к файлам, сниппеты, слова из открытых буферов.

| Хоткей | Где | Действие |
|---|---|---|
| `<Tab>` | Insert | Если меню открыто — выбрать выделенный вариант и принять. Если активен сниппет — прыгнуть на следующий placeholder. Иначе — обычный Tab |
| `<S-Tab>` | Insert | Назад по сниппету / предыдущий вариант в меню |
| `<CR>` | Insert | Принять выбранный вариант |
| `<C-Space>` | Insert | Открыть меню вручную (если само не всплыло) |
| `<C-n>` / `<C-p>` | Insert | Следующий / предыдущий вариант в меню |
| `<C-e>` | Insert | Закрыть меню |
| `<C-b>` / `<C-f>` | Insert | Скролл окна документации (превью справа от меню) |

### Сниппеты

Сниппет — заготовка кода с «дырками» (placeholder), по которым прыгает курсор. В меню автодополнения сниппеты помечены отдельным значком.

Как использовать: начни набирать имя сниппета (например в Go — `forr`, `fori`, `iferr`), выбери его из меню, нажми `<Tab>`. Код развернётся, курсор встанет на первый placeholder. `<Tab>` — прыжок к следующему, `<S-Tab>` — назад.

| Пример (Go) | Что разворачивает |
|---|---|
| `forr` | `for i, v := range ... {}` |
| `fori` | `for i := 0; i < ...; i++ {}` |
| `iferr` | `if err != nil { return ... }` |

Полный список сниппетов зависит от языка — friendly-snippets покрывает Go, Lua, YAML, JSON, Dockerfile, Markdown и другие.

### Авто-скобки

При выборе функции из меню скобки добавляются автоматически: набрал `Println` → `<Tab>` → получилось `Println()` с курсором внутри.

## 20. Форматирование и линтинг

**Форматирование** — conform.nvim. **Линтинг** — nvim-lint. Оба дополняют LSP: conform управляет форматированием единообразно для всех языков, nvim-lint добавляет внешние линтеры, которых нет в LSP.

### Форматирование (conform.nvim)

| Хоткей / команда | Действие |
|---|---|
| `<leader>cf` | Форматировать документ или выделение вручную (работает в Normal и Visual) |
| *(автоматически)* | **Format-on-save** — файл форматируется при каждом `:w` |
| `:FormatToggle` | Включить / выключить format-on-save **глобально** (для всех буферов) |
| `:FormatToggle!` | То же, но **только для текущего буфера** (с `!`) |

Для Go форматирование идёт цепочкой: `goimports` (сортирует импорты, добавляет недостающие, убирает неиспользуемые) → `gofumpt` (строже стандартного gofmt по оформлению).

### Линтинг (nvim-lint)

Линтеры запускаются автоматически при открытии файла, при сохранении и при выходе из Insert mode. Их вывод попадает в ту же систему диагностик, что и LSP — те же squiggles, `]d`/`[d`, панель Trouble.

| Хоткей / команда | Действие |
|---|---|
| `<leader>ll` | Запустить линтинг текущего файла вручную |
| `:LintInfo` | Показать, какие линтеры назначены текущему типу файла |

| Язык | Линтер | Что ловит |
|---|---|---|
| Go | `golangci-lint` | металинтер, ~50 линтеров: проигнорированные ошибки (`errcheck`), мёртвый код (`unused`), бесполезные присваивания (`ineffassign`) и т.д. |
| Dockerfile | `hadolint` | best practices: незапиненные версии, запуск от root, `apt-get` без очистки кэша |

**Важно про Go:** `golangci-lint` v2 без файла `.golangci.yml` в корне проекта запускает лишь минимальный набор проверок. Для полноценного линтинга в проекте нужен `.golangci.yml` с `version: "2"` и секцией `linters.enable`.

## 21. Trouble — панель проблем

Аналог панели **Problems** в VS Code (`Ctrl+Shift+M`). Открывается отдельным окном внизу со списком проблем — можно ходить по ним и прыгать к нужному месту в коде.

| Хоткей | Действие |
|---|---|
| `<leader>xx` | список **ошибок и предупреждений всего проекта** (то, что LSP-серверы нашли) |
| `<leader>xd` | то же, **только текущий файл** |
| `<leader>xt` | список всех `TODO` / `FIXME` / `HACK` в проекте |
| `<leader>xq` | показать в красивом виде **quickfix list** (см. ниже) |
| `<leader>xl` | показать **location list** текущего окна (см. ниже) |

### Внутри панели Trouble

| Хоткей | Действие |
|---|---|
| `<CR>` | прыгнуть в код к выбранной проблеме |
| `j` / `k` | вниз / вверх |
| `o` | открыть в коде, не теряя фокус |
| `q` | закрыть панель |
| `?` | встроенная справка по биндингам |

### Про quickfix и location list

Это два встроенных в Vim «блокнотика» со списком мест в файлах. Туда что-то попадает только когда специальная команда или плагин решил это положить.

- **Quickfix** (`<leader>xq`) — один на всё Neovim. Заполняется командами `:grep`, `:make`, или плагинами (например, todo-comments при `<leader>xt` параллельно складывает туда же).
- **Location list** (`<leader>xl`) — свой у каждого окна. Заполняется командами `:lgrep`, `:lvimgrep`. На практике в нашей работе **почти не используется** — оставлен на будущее.

Обычно тебе нужны только **`<leader>xx`**, **`<leader>xd`**, **`<leader>xt`**.

## 24. Terminal — встроенный терминал

Терминал внутри Neovim через `snacks.terminal`. Внутри каждого — полноценный **fish** со всей твоей конфигурацией, алиасами и историей. Отличие от отдельного окна терминала: окном управляет Neovim — его можно прятать/показывать одной клавишей, а из вывода прыгать в код.

**Четыре терминала, у каждого своя роль.** Роль — это соглашение для удобства, технически в любом терминале можно делать что угодно (везде одинаковый fish).

| Хоткей | Терминал | Позиция | Роль |
|---|---|---|---|
| `<C-/>` | term-shell | сплит снизу (1/3 высоты) | основной рабочий: `go build`, `go get`, `go test` руками, git |
| `<leader>Tf` | term-float | плавающее окно поверх кода | короткие разовые команды: «вызвал — посмотрел — закрыл» |
| `<leader>Tr` | term-run | сплит снизу (1/3 высоты) | запущенный сервер: `go run ./cmd/...`, отдельная сессия для логов |
| `<leader>Tw` | term-watch | сплит сбоку (1/3 ширины) | под `air` (hot-reload) — оживёт в Блоке 14 |

`<C-/>` работает и в режиме вставки (можно вызвать прямо из набора кода). `<C-_>` — алиас на тот же term-shell (некоторые терминалы шлют `<C-/>` именно так).

### Открыть / свернуть / закрыть

| Действие | Как | Что происходит |
|---|---|---|
| Открыть / свернуть | та же клавиша терминала (toggle) | свернуть = спрятать окно, **процесс внутри жив** (сервер продолжает работать) |
| Выйти из печати в normal-режим | `<Esc><Esc>` | можно листать вывод моторами Neovim: `j`/`k`, поиск `/`, копирование |
| Вернуться к печати | `i` или `a` | снова terminal-режим |
| Остановить процесс внутри | `Ctrl+C` (в terminal-режиме) | убивает сервер/команду, fish остаётся |
| Закрыть терминал, мягко | `exit` + `<CR>` (в fish) | fish завершается, окно само закрывается (`auto_close`) |
| Закрыть терминал, жёстко | `<Esc><Esc>`, затем `:bdelete!` | убивает буфер вместе с процессом разом |

**Свернуть vs закрыть:** свернуть (toggle-клавиша) — процесс жив, развернёшь — всё на месте. Закрыть (`exit` / `:bdelete!`) — процесс убит, при следующем toggle терминал создаётся заново с чистого листа.

### Несколько терминалов одновременно

Сплиты в одной позиции **складываются стопкой**. Открыл `<leader>Tr` (сервер) и `<C-/>` (term-shell) — оба внизу, переключение между ними — `<C-w>` затем `j` / `k`. Типовой сценарий: в `term-run` крутится сервер, в `term-shell` рядом гоняешь `curl` — оба видны без сворачивания.

### Метка терминала

В каждом терминале переменная `$NVIM_TERM` хранит его имя (`shell` / `float` / `run` / `watch`) — `echo $NVIM_TERM` покажет, где ты находишься. Через неё же терминалы технически различаются между собой.

## 25. Git — gitsigns, lazygit, diffview

Три инструмента, каждый закрывает свой слой работы с git:

- **gitsigns** — git прямо в буфере с кодом: знаки изменений на полях, точечный stage/reset кусков, inline-blame. Не выходя из редактирования.
- **lazygit** — отдельное окно-комбайн для коммитов, веток, rebase, stash, log. Аналог GitLens + Source Control в VS Code, только клавиатурой.
- **diffview** — полноэкранный side-by-side diff и история коммитов файла/репозитория.

### gitsigns — знаки на полях

Слева от номеров строк (в gutter) появляются знаки относительно git-индекса:

| Знак | Значение |
|---|---|
| `▎` (рядом с номером) | строка добавлена или изменена |
| `▁` / `▔` | строки удалены ниже / выше |
| `┆` | файл не отслеживается git (untracked) |

Знаки для уже застейдженных (`git add`) кусков рисуются отдельным цветом — видно, что готово к коммиту, а что ещё нет.

### gitsigns — навигация и операции с hunk'ами

**Hunk** — связанный блок изменённых соседних строк, атомарная единица стейджинга (как `git add -p`).

| Хоткей | Действие | VS Code-аналог |
|---|---|---|
| `]h` / `[h` | следующий / предыдущий hunk | навигация в git-панели |
| `<leader>ghp` | превью hunk'а во всплывающем окне | наведение на ⏵ слева |
| `<leader>ghs` | **stage hunk** (= `git add` этого куска) | `+` на hunk'е |
| `<leader>ghr` | **reset hunk** (откатить изменение) | ↶ Revert Change |
| `<leader>ghu` | undo последнего stage | — |
| `<leader>ghS` | stage **всего файла** (заглавная S) | Stage Changes |
| `<leader>ghR` | reset **всего файла** (заглавная R) | Discard Changes |
| `<leader>ghd` | diff файла против индекса | — |
| `<leader>ghD` | diff файла против `HEAD~` (заглавная D) | — |
| `<leader>ghb` | полный blame строки во всплывающем окне | GitLens blame |

В **Visual mode** `<leader>ghs` / `<leader>ghr` стейджат/откатывают именно выделенные строки, а не весь hunk целиком.

### gitsigns — inline blame

## 27. Debug — nvim-dap

Отладка Go-кода прямо в редакторе: точки останова, пошаговое выполнение, просмотр живых значений переменных, стек вызовов, REPL. Аналог вкладки **Run and Debug** в VS Code. Под капотом — тот же отладчик `delve` (`dlv`), что использует и расширение Go в VS Code.

Связка из четырёх плагинов: **nvim-dap** (движок протокола DAP), **nvim-dap-go** (мост к `delve` + Go-конфигурации запуска), **nvim-dap-ui** + **nvim-nio** (графические панели). Грузится лениво — при первом нажатии любой `<leader>d*`.

### Хоткеи

| Хоткей | Действие |
|---|---|
| `<leader>db` | поставить / убрать **breakpoint** (точку останова) на текущей строке |
| `<leader>dc` | **continue**: запустить отладку, либо продолжить до следующей точки останова |
| `<leader>do` | **step over**: выполнить строку целиком, не заходя внутрь вызовов |
| `<leader>di` | **step into**: зайти внутрь вызываемой функции |
| `<leader>dO` | **step out**: доисполнить текущую функцию и выйти в вызвавшую (заглавная O) |
| `<leader>dt` | **terminate**: принудительно завершить сессию отладки |
| `<leader>dr` | toggle **REPL** (интерактивный ввод) |
| `<leader>du` | toggle всех панелей dap-ui |
| `<leader>de` | показать значение переменной под курсором во всплывающем окне (Normal и Visual) |

### Запуск: меню конфигурации delve

После `<leader>dc` (если сессии ещё нет) delve спрашивает, **что** запускать. Набери цифру + `<Enter>`:

| Пункт | Когда выбирать |
|---|---|
| `1: Debug` | обычная программа с `func main` — **стандартный случай** |
| `2: Debug (Arguments)` | программе нужны аргументы командной строки (`os.Args`) |
| `3: Debug (Arguments & Build Flags)` | + флаги сборки, например `-tags` для `//go:build` |
| `4: Debug Package` | отлаживать весь пакет (папку), а не текущий файл |
| `5: Attach` | подключиться к уже запущенному процессу (контейнер и т.п.) |
| `6: Debug test` | отладить тест, ближайший к курсору |
| `7: Debug test (go.mod)` | то же, но контекст запуска — корень модуля |

### Панели dap-ui (раскладка: Вариант 3)

При остановке на точке экран перестраивается: **правая колонка** + **нижняя полка**. neo-tree остаётся слева на месте.

| Панель | Где | Что показывает |
|---|---|---|
| **Scopes** (`Locals`) | правая колонка | все локальные переменные текущей точки — обновляются на каждом шаге |
| **Breakpoints** | правая колонка | список всех точек останова по проекту (`<CR>` — прыгнуть к точке) |
| **Stacks** | правая колонка | стек вызовов: кто кого вызвал (`<CR>` — прыгнуть на уровень) |
| **REPL** | нижняя полка | интерактивный ввод: `i`, выражение, `<CR>` — узнать значение |
| **Watches** | нижняя полка | свои выражения слежения — встань в панель, `i`, выражение, `<CR>` |

Стрелка-указатель и подсветка строки в коде = «программа сейчас здесь». Ряд кликабельных кнопок (`controls`) отключён намеренно — управление только клавишами.

### Важные нюансы

- **`<leader>dc` в цикле.** Если точка останова стоит внутри цикла, continue упрётся в неё на **каждом витке** — программа не «добежит до конца». Чтобы закрыть сессию: `<leader>dt`, либо убрать точку (`<leader>db`) и тогда `<leader>dc`.
- **Сессия закрывается сама**, когда программа доходит до конца, — панели убираются автоматически.
- **Съехали окна** после сворачивания/закрытия других окон ОС поверх kitty — набери руками `:wincmd =`, окна выровняются.
